# Documentation Audit Report

## Part 1: Findings by File

### `docs/technical/architecture.md`

- **Consistency** — Section "Performance > No API caching" (line 152) states: *"Every navigation or action that needs TMDB data makes a fresh API request. There is no response cache, no request deduplication, and no stale-while-revalidate layer."* This directly contradicts `data-model.md` line 149, which describes `useGenres()` as *"Fetches movie and TV genre lists from TMDB on first call and caches the result in memory for the session."* **Fix:** Add a carve-out in the architecture doc acknowledging that `useGenres()` is the one exception to the no-caching rule, or adjust the data-model description.

- **Correctness** — The component hierarchy (line 179) shows `TrendingCarousel → MovieCard[]`, implying the trending carousel renders movie cards. However, `ui-ux.md` section 5 titles its carousel spec as *"Hero Banner (Trending Carousel)"* and describes full-width backdrop banners — a fundamentally different component from `MovieCard`. **Fix:** Update the hierarchy to show `TrendingCarousel → HeroBanner[]` (or similar), consistent with the UI spec.

- **Correctness** — The component hierarchy for `EntryDetails` (lines 185–191) does not include a gallery component, yet `ui-ux.md` section 5 "Detail Screen" specifies *"Gallery — Horizontal row of thumbnail images at the bottom"* and `conventions.md` section 11 defines image sizing for *"Gallery thumbnails"*. **Fix:** Either add a gallery component to the hierarchy or remove gallery references from `ui-ux.md` and `conventions.md`.

### `docs/technical/ui-ux.md`

- **Consistency** — Section 3 "Layout" breakpoint table: `max-sm` (< 640px) lists **2** columns but the context column says *"Single-column mobile layout"*. Two columns is not single-column. **Fix:** Change the context to *"Two-column mobile layout"* or change the column count to 1.

- **Consistency** — Section 3 "Layout" breakpoint table: `max-lg` (< 1024px) context says *"Sidebar appears, multi-column"*. The sidebar should already be visible at all larger breakpoints (per conventions.md: *"md and above: sidebar navigation"*). Saying it "appears" at `max-lg` implies it wasn't present before. **Fix:** Change to *"Sidebar visible, multi-column"* or similar.

- **Clarity** — Section 5 "Hero Banner" mentions an *"In Theaters Now >"* text link on the right side of the banner without explaining its navigation destination. **Fix:** Specify where the link navigates (presumably `/calendar` for the Release Calendar).

- **Completeness** — The Stats screen has components defined in `architecture.md` (`StatCards`, `GenreChart`, `MonthlyChart`, `TopRatedList`) and a composable in `data-model.md` (`useStats()`), but `ui-ux.md` has no design section for it — only an empty-state row. **Fix:** Add a Stats screen design section.

- **Completeness** — The Recommendations screen has a component hierarchy in `architecture.md` and composable in `data-model.md`, but `ui-ux.md` has no design section — only an empty-state row. **Fix:** Add a Recommendations screen design section.

- **Completeness** — The Release Calendar screen has a component hierarchy in `architecture.md` (`CalendarGrid → ReleaseCard[]`), but `ui-ux.md` has no design section — only an empty-state row. **Fix:** Add a Release Calendar screen design section.

- **Completeness** — The Settings screen has a component hierarchy in `architecture.md` (`ThemeToggle`, `LanguageSelect`, `RegionSelect`, `HomeSectionSelect`), but `ui-ux.md` section 12 only briefly mentions theme support. No full settings screen layout or interaction design exists. **Fix:** Add a Settings screen design section.

### `docs/technical/setup.md`

- **Accuracy** — The "Commands" table (lines 31–37) lists `npm run dev`, `npm run build`, `npm run preview`, `npm run lint`, and `npm run test` as if they are currently available, but no `package.json` exists in the project. Running `npm install` (line 11) would also fail. **Fix:** Add a note that these commands will become available after the project is initialized (e.g., after running `npm create vue@latest` or scaffolding `package.json`).

### `docs/technical/deployment.md`

- **Accuracy** — The doc describes `firebase.json` and `.firebaserc` as generated by `firebase init hosting` (lines 26–27). While the generation step is documented, neither file currently exists in the repo, yet the doc reads as if the deployment pipeline is set up. **Fix:** Add a note clarifying that Firebase Hosting has not yet been initialized and these files will be created when `firebase init hosting` is run.

### `docs/technical/testing.md`

- **Cross-Reference** — Testing rules are also covered in `conventions.md` section 6. Neither document links to the other. A developer reading only one doc may miss relevant guidance in the other. **Fix:** Add a cross-reference from `testing.md` to `conventions.md` section 6 (and vice versa).

### `docs/technical/conventions.md`

- **Cross-Reference** — Section 6 "Testing" covers testing conventions that overlap with `testing.md`. No cross-reference to `testing.md` is provided. **Fix:** Add a link to `testing.md` for the full testing specification.

- **Clarity** — Section 10 "Internationalization" states the app sends ISO 639-1 codes (e.g., `"en"`), while `api.md` endpoint tables consistently show `en-US` as the default `language` parameter value. While the text explains that TMDB accepts both formats, the visual discrepancy between what the app sends (`"en"`) and what the API tables show (`en-US`) could confuse developers implementing the API client. **Fix:** Add a note in the `api.md` endpoint tables clarifying that the app will send `"en"` (not `"en-US"`) per the i18n convention, or unify the format across docs.

### General (across all docs)

- **Clarity** — Tense is inconsistent across documentation. Most files use present tense as if describing an existing system (*"The app uses..."*, *"Routes are defined in..."*) while a few use future tense (*"All types will be defined..."* in `api.md` line 84 and `data-model.md` line 1). Since no source code exists yet, the mixed tense can mislead new readers about the project's state. **Fix:** Choose one approach consistently — either present tense throughout (design-doc style, describing intended behavior) or future tense. If using present tense, add a note at the top of `docs/index.md` stating these are specifications for a system not yet implemented.

## Part 2: Missing Documentation

- **Stats screen design** — `ui-ux.md` lacks a design section for the Stats screen. The architecture defines four sub-components (`StatCards`, `GenreChart`, `MonthlyChart`, `TopRatedList`) with no visual specification for what each displays, their layout, chart types, or data presentation.

- **Recommendations screen design** — `ui-ux.md` lacks a design section for the Recommendations screen. The architecture defines a `FilterBar` and `EntryGrid` but doesn't specify how seed entries are surfaced to the user, how recommendations are grouped, or what filter options are available.

- **Release Calendar screen design** — `ui-ux.md` lacks a design section for the Release Calendar. The architecture defines `CalendarGrid → ReleaseCard[]` but doesn't specify the calendar layout (month view, week view, list), date navigation, or `ReleaseCard` visual design.

- **Settings screen design** — `ui-ux.md` lacks a full design section for the Settings screen. The architecture hierarchy shows four controls (`ThemeToggle`, `LanguageSelect`, `RegionSelect`, `HomeSectionSelect`) but their layout, interaction patterns, and visual styling are unspecified.

- **Data export/import** — The routing table in `architecture.md` says the Settings screen includes *"Theme, language, data export"*, but no documentation describes data export functionality — not in the data model, composables, or UI spec. Either document it or remove the reference.
